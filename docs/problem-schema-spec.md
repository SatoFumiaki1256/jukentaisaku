# 課題データ仕様書 — Problem JSON Schema Specification

**バージョン**: 1.0  
**対象**: 課題メーカー（管理者ツール）↔ エスキスアプリ間のデータ受け渡し

---

## 1. 概要

課題データは **JSON形式** で管理し、`problems.json`（配列）として配信する。  
管理者ツールでは 1課題 = 1オブジェクトを生成・編集し、アプリ側はこの配列を読み込んで表示・動作する。

```
[管理者ツール] → problems.json（配列）→ [エスキスアプリ]
```

---

## 2. トップレベル構造

```jsonc
[
  { /* Problem オブジェクト */ },
  { /* Problem オブジェクト */ },
  ...
]
```

`problems.json` は **Problem オブジェクトの配列**。順序がそのまま表示順になる。

---

## 3. Problem オブジェクト — フィールド定義

### 3.1 基本情報

| フィールド | 型 | 必須 | 説明 | 例 |
|---|---|---|---|---|
| `id` | `number` | ✅ | 一意の整数ID（連番推奨） | `6` |
| `title` | `string` | ✅ | 課題タイトル（表示名） | `"シェアハウス（共同居住型賃貸住宅）"` |
| `desc` | `string` | — | 課題の概要説明（カード表示・詳細ページ） | `"ある地方都市において..."` |
| `difficulty` | `"基礎"｜"応用"｜"実践"` | ✅ | 難易度バッジ | `"実践"` |

> **difficulty の色分け**:  
> `基礎` → 青、`応用` → 黄、`実践` → 赤

---

### 3.2 (1) 敷地条件

| フィールド | 型 | 必須 | 説明 | 例 |
|---|---|---|---|---|
| `siteW` | `number` | ✅ | 敷地幅（mm単位、455の倍数推奨） | `16380` |
| `siteH` | `number` | ✅ | 敷地奥行（mm単位、455の倍数推奨） | `18200` |
| `zoning` | `string` | ✅ | 用途地域 | `"第一種住居地域"` |
| `zoningSlope` | `number` | — | 北側斜線勾配 | `1.25` |
| `fireZone` | `string` | — | 防火指定 | `"指定なし"` |
| `coverage` | `number` | ✅ | 建蔽率（%） | `60` |
| `far` | `number` | ✅ | 容積率（%） | `200` |
| `coverageBonus` | `string` | — | 建蔽率加算の根拠（角地等） | `"特定行政庁が指定した角地における加算を含む"` |
| `terrain` | `string` | — | 地形 | `"平坦"` |

#### siteAdj（敷地隣接条件）

```jsonc
"siteAdj": {
  "N": { "type": "宅地" },
  "S": { "type": "道路", "width": 6 },
  "E": { "type": "道路", "width": 4 },
  "W": { "type": "宅地" }
}
```

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `type` | `string` | ✅ | 隣接用途（後述の選択肢） |
| `width` | `number` | 道路時✅ | 幅員（m） |
| `opposite` | `string` | — | 対面の用途（道路の場合のみ） |
| `oppositeWidth` | `number` | — | 対面の幅（m） |

**type 選択肢**:  
`道路` / `宅地` / `公園` / `緑地` / `水路` / `河川` / `空地` / `広場` / `里道` / `線路敷`

---

### 3.3 (2) 構造・建築物の高さ等

| フィールド | 型 | 必須 | 説明 | 例 |
|---|---|---|---|---|
| `structure` | `string` | ✅ | 構造種別 | `"木造2階建"` |
| `floors` | `number` | — | 階数 | `2` |
| `maxHeight` | `number` | — | 最高の高さ（m） | `10` |
| `eavesHeight` | `number` | — | 軒の高さ（m） | `7` |
| `structureNotes` | `string[]` | — | 構造に関する追加条件 | `["耐力壁は..."]` |

---

### 3.4 (3) 設計上の留意事項

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `designConditions` | `string[]` | — | ①②③形式で表示される設計留意事項（最大5件推奨） |

```jsonc
"designConditions": [
  "入居者の交流とプライバシーの確保に配慮する",
  "上下階の床衝撃音・騒音等に配慮する",
  "LDKについては、耐震性及び上階からの荷重等に配慮する"
]
```

---

### 3.5 (4) 延べ面積等

| フィールド | 型 | 必須 | 説明 | 例 |
|---|---|---|---|---|
| `floorAreaMin` | `number` | — | 延べ面積の下限（㎡） | `200` |
| `floorAreaMax` | `number` | — | 延べ面積の上限（㎡） | `250` |
| `floorAreaExclusion` | `string` | — | 床面積算入除外対象の注記 | `"バルコニー、ポーチ等は..."` |
| `notes` | `string` | — | その他の補足メモ（改行`\n`可） | `"1階：LDK・水回り\n2階：..."` |

> **注意**: `floorAreaMin` / `floorAreaMax` どちらか片方だけでも有効。  
> 旧形式の `floorArea`（単一値）も後方互換として読み込み可能。

---

### 3.6 (5) 要求室等（reqRooms）

```jsonc
"reqRooms": [
  {
    "name": "LDK",
    "floor": "1F",
    "category": "共用部分",
    "qty": 1,
    "area": "25㎡以上",
    "notes": [
      "1室にまとめ、25㎡以上として計画する",
      "屋外テラスと直接行き来できる出入口を設ける"
    ]
  }
]
```

| フィールド | 型 | 必須 | 説明 | 例 |
|---|---|---|---|---|
| `name` | `string` | ✅ | 室名 | `"LDK"`, `"個室（A）"` |
| `floor` | `string` | ✅ | 設置階（後述の選択肢） | `"1F"` |
| `category` | `string` | — | 部分分類（グループ表示に使用） | `"共用部分"` |
| `qty` | `number` | — | 室数（省略時=1室） | `4` |
| `area` | `string` | — | 面積条件（省略時=「適宜」表示） | `"13㎡以上"` |
| `notes` | `string[]` | — | 特記事項（ア・イ・ウ形式で表示） | `["下足入れを設ける", ...]` |

**floor 選択肢**:  
`"1F"` / `"2F"` / `"各階"` / `"どちらでも"`

**category 推奨値（自由記述も可）**:  
`"共用部分"` / `"専用部分"` / `"居住部分"` / `"水回り"` / `"収納"` /  
`"親世帯"` / `"子世帯"` / `"店舗部分"` / `"住宅部分"` / `"地域活動部分"`

> 要求室の**表示順 = 配列順**。  
> floor → category の順でグルーピングされて表示される。  
> 同じfloor内で category が切り替わると小見出しが挿入される。

---

### 3.7 (6) 屋外施設等（exteriorConds）

```jsonc
"exteriorConds": [
  {
    "type": "屋外テラス",
    "notes": [
      "LDKと直接行き来できるようにする",
      "テーブル及び椅子（計6席以上）を設ける"
    ]
  },
  {
    "type": "駐輪スペース",
    "notes": ["10台分を設ける"]
  },
  {
    "type": "門・塀・植栽等",
    "notes": []
  }
]
```

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `type` | `string` | ✅ | 施設名 |
| `notes` | `string[]` | — | 特記事項（ア・イ・ウ形式） |

## 6. バリデーションルール

管理者ツール側で保存時にチェックすべき項目：

| チェック項目 | ルール |
|---|---|
| `id` | 正の整数、配列内で一意 |
| `title` | 空でないこと |
| `difficulty` | `基礎` / `応用` / `実践` のいずれか |
| `siteW`, `siteH` | 1820以上、455の倍数（グリッド整合）|
| `coverage` | 1〜100の整数 |
| `far` | 1以上の整数 |
| `floorAreaMin` | floorAreaMax より小さいこと |
| `siteAdj.*.type` が `道路` | `width` が必須 |
| `reqRooms[*].name` | 空でないこと |
| `reqRooms[*].floor` | 選択肢のいずれか |
| `designConditions` | 最大5件推奨 |
| `reqRooms[*].notes` | 最大10件推奨 |

## 7. アプリ側の読み込み挙動（参考）

エスキスアプリ（`app.html`）は課題を以下のように利用する：

```
problems.json
  ↓ fetch（cache:no-store）
index.html  → カード一覧表示
  ↓ クリック
problem-detail.html?id=X  → 課題詳細表示
  ↓ 「エスキスを始める」
app.html?problemId=X  → エスキス画面（課題条件を自動ロード）
```

| appの動作 | 参照フィールド |
|---|---|
| 敷地サイズをキャンバスに反映 | `siteW`, `siteH` |
| 条件パネルのフォームを埋める | `title`, `structure`, `zoning`, `coverage`, `far`, `floorAreaMin`, `floorAreaMax`, `notes` |
| 隣接条件を設定 | `siteAdj` |
| 要求室リストに追加 | `reqRooms[].name`, `.floor`, `.area`, `.notes`, `.qty`, `.category` |
| 外構条件リストに追加 | `exteriorConds[].type`, `.notes` |
| チェックリスト検証 | `floorAreaMin`, `floorAreaMax`, `reqRooms` |

│                                     │

│   最小 / 最大 / 除外注記            │
│                                     │

## 9. エクスポート形式

### 9.1 単一課題エクスポート
```json
{ /* Problem オブジェクト 1件 */ }
```

### 9.2 複数課題バンドル
```json
[
  { /* Problem */ },
  { /* Problem */ }
]
```

エスキスアプリへの取り込み方法：
- `problems.json` を置き換えるか追記する
- または API エンドポイントでマージする
